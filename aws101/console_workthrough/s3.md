## Kiro Grade: B-

### Per-section feedback

**1. AWS Region** ✅ Correct.
- S3 buckets are regional. Good to note.
- One addition: once created, you can't change a bucket's region.

**2. Bucket namespace** ⚠️ Partially correct, some confusion.
- Two options are: globally unique name (default, traditional) vs account+region scoped name (newer option).
- "api call로 부를 수 있어 무관한 사람이 s3에 요청이 가능하다" — this is about the old global namespace where anyone could guess your bucket name and send requests. The newer account-scoped namespace helps prevent this.
- The billing incident you mentioned is a real thing — correct. Third parties could make requests to your bucket URL and you'd get charged for the denied requests. AWS has since addressed this.

**3. Copy settings from existing bucket** ✅ Correct concept.
- It copies settings (encryption, versioning, public access, etc.) from an existing bucket to pre-fill the form. It does NOT copy the bucket policy. Console-only feature (not available in CLI).

**4. Object Ownership / ACL** ❌ Major confusion.
- You're mixing up S3 ACLs with VPC Network ACLs (NACLs). They are completely different things:
  - **S3 ACL** = access control for S3 objects/buckets (who can read/write objects). Legacy feature.
  - **VPC NACL** = network-level firewall for subnets (stateless, inbound/outbound rules). This is what you described.
  - **Security Group** = instance-level firewall (stateful).
- S3 Object Ownership has 3 options:
  - **Bucket owner enforced** (default, recommended) — ACLs disabled, bucket owner owns everything, use policies for access control
  - **Bucket owner preferred** — ACLs enabled, bucket owner gets ownership if uploader uses `bucket-owner-full-control` ACL
  - **Object writer** — ACLs enabled, the uploader owns the object
- AWS recommends keeping ACLs disabled (bucket owner enforced) because bucket policies are simpler and more flexible.

**5. Block Public Access** ✅ Correct.
- Default is all 4 settings blocked. Recommended to keep all enabled unless you specifically need public access (e.g. static website hosting).

**6. Bucket Versioning** ✅ Mostly correct.
- Delete marker concept is correct.
- Previous versions are kept and do incur storage costs — correct.
- Minor clarification: deleted objects aren't "eventually deleted" automatically. The delete marker hides the object, but old versions remain forever unless you set up a Lifecycle policy to clean them up.

**7. Tags** ✅ Fine.

**8. Default encryption** ⚠️ Incomplete but honest.
- Three options:
  - **SSE-S3** (default) — AWS manages the keys entirely. Free. No extra config needed.
  - **SSE-KMS** — AWS KMS manages the keys. You can use the AWS-managed key (`aws/s3`, free) or your own customer-managed key (costs per API call). Gives you key rotation, audit trail via CloudTrail, and fine-grained access control.
  - **DSSE-KMS** — Double encryption with two separate KMS keys. For highly regulated workloads (e.g. CNSSP 15 compliance). Rarely needed.
- The key difference: SSE-S3 = simple/free, SSE-KMS = more control + audit + cost.

### Missing console options
- **Advanced settings → S3 Object Lock** — prevents objects from being deleted or overwritten for a retention period. Automatically enables versioning. Used for compliance (WORM — Write Once Read Many).
