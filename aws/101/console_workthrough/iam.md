##### Policies - Specify permissions

- 네가 쓴 `이러한 정책을 만드는 이점은 잘 모르겠다`의 핵심은, policy를 `권한 묶음` 정도로만 보고 있다는 점이다.
- 더 정확히는 policy는 `누가 어떤 AWS API를 어떤 리소스에 대해 할 수 있는가`를 적는 권한 문서다.
- 예를 들어:
  - `Action` = `s3:GetObject`
  - `Resource` = `arn:aws:s3:::my-bucket/*`
  - `Effect` = `Allow`
- 즉 policy 생성의 이점은:
  - 권한을 반복해서 재사용 가능
  - user/group/role에 같은 정책을 붙일 수 있음
  - 최소 권한 원칙을 문서 수준에서 명확히 설계 가능
  - JSON으로 세밀한 제어가 가능

##### Roles - Select trusted entity

- trusted entity는 `누가 이 role을 쓸 수 있는가`를 정하는 단계다.
- permissions policy는 `그 role로 무엇을 할 수 있는가`를 정하는 단계다.
- 이 둘은 반드시 분리해서 이해해야 한다.
- 예시:
  - `EC2`를 trusted entity로 선택
  - `AmazonS3ReadOnlyAccess`를 permission으로 부여
  - 의미: `EC2 인스턴스가 이 role을 assume해서 S3 읽기를 할 수 있다`

##### Identity providers

- 네가 쓴 `3자가 보안/인증을 거치면 aws에서도 적용되는 단계인지 궁금하다`는 방향은 맞다.
- 정확히는 외부 인증 시스템이 사용자를 인증하고, AWS는 그 인증 결과를 신뢰해서 IAM role을 부여한다.
- 이것을 `federation`이라고 한다.
- 그래서 회사가 Okta, Azure AD, Google Workspace 같은 중앙 인증 체계를 쓰면 AWS에 IAM user를 많이 만들지 않고도 접근을 줄 수 있다.

##### Audience

- `Audience`는 `이 토큰이 누구를 대상으로 발급된 것인가`를 확인하는 값이다.
- AWS는 OIDC 토큰을 받을 때 `이 토큰이 정말 AWS용인지`, `내가 신뢰한 애플리케이션용인지`를 확인해야 한다.
- 쉽게 말하면 `이 로그인 티켓이 진짜 이 서비스 앞으로 발급된 것인지 확인하는 값`이다.
- 그래서 단순히 `어떤 사용자인지 지정`하는 것이 아니라, `이 토큰의 대상 시스템이 맞는지` 검증하는 데 가깝다.

##### IAM User, Group, Role은 어떻게 다른가?

- `IAM User`는 사람이나 애플리케이션을 위한 장기적인 AWS 주체다.
- `Group`은 여러 user를 묶어서 같은 권한을 관리하기 위한 단위다.
- `Role`은 누군가가 임시로 assume해서 쓰는 권한 단위다.

##### 왜 EC2에는 access key 대신 role을 붙이는가?

- access key는 장기 자격 증명이어서 서버 안에 남으면 유출 위험이 크다.
- role을 붙이면 EC2가 임시 자격 증명을 자동으로 받아 사용한다.
- 그래서 AWS는 EC2, Lambda 같은 서비스에 role 사용을 권장한다.

##### managed policy와 customer managed policy는 어떻게 다른가?

- `AWS managed policy`는 AWS가 미리 만든 정책이다.
- `Customer managed policy`는 사용자가 직접 만든 정책이다.
- 후자가 최소 권한 설계를 더 세밀하게 하기 좋다.

##### inline policy와 managed policy는 어떻게 다른가?

- `Inline policy`는 특정 user, group, role 안에 직접 붙는 정책이다.
- `Managed policy`는 독립된 정책 객체라서 여러 대상에 재사용할 수 있다.

##### MFA는 왜 중요한가?

- 비밀번호가 유출되어도 추가 인증 수단이 필요하게 만든다.
- 특히 콘솔 로그인 계정과 권한이 큰 계정에는 매우 중요하다.

##### access key는 언제 만들고, 왜 위험한가?

- CLI, SDK, API 호출처럼 프로그래밍 방식 접근이 필요할 때 만든다.
- 하지만 장기 자격 증명이므로 코드나 서버에 남으면 유출 위험이 크다.
- 가능하면 role이나 federation을 먼저 고려하는 것이 좋다.

##### cross-account role은 언제 쓰는가?

- 다른 AWS 계정의 사용자나 서비스가 내 계정 리소스에 접근해야 할 때 쓴다.
- 새 user를 만드는 대신 role assume 방식으로 권한을 위임하는 구조다.

##### federation과 IAM user 방식은 어떻게 다른가?

- `IAM user`는 AWS 안에 사용자를 직접 만드는 방식이다.
- `Federation`은 외부 인증 시스템이 사용자를 인증하고, AWS는 그 결과를 신뢰해 role을 부여하는 방식이다.
- 대규모 조직이나 회사 SSO 환경에서는 federation이 더 일반적이다.
