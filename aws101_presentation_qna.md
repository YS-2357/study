# AWS 101 발표 예상질문

## 1. 전체 발표 공통 질문

### Q1. 클라우드 컴퓨팅이 온프레미스보다 좋은 이유는 무엇인가요?
- 왜 물을까: 발표 초반의 핵심 메시지를 다시 확인하려는 질문
- 짧은 답:
  - 초기 투자비용(CAPEX)을 줄이고 사용한 만큼만 비용을 내는 구조로 운영할 수 있습니다.
  - 필요할 때 빠르게 자원을 늘리거나 줄일 수 있어서 민첩성과 확장성이 높습니다.
- 더 들어오면:
  - 모든 시스템에 무조건 클라우드가 정답은 아니고, 규제, 레거시 구조, 예측 가능한 고정 워크로드는 온프레미스가 더 유리할 수도 있습니다.

### Q2. 왜 굳이 AWS를 써야 하나요?
- 왜 물을까: “클라우드는 이해했는데 왜 AWS냐”를 확인하려는 질문
- 짧은 답:
  - AWS는 서비스 종류가 매우 많고 글로벌 인프라가 넓으며, 다양한 운영 경험이 축적된 플랫폼입니다.
  - 초보자 입장에서도 필요한 서비스를 조합해서 빠르게 시작하기 좋습니다.
- 더 들어오면:
  - 실제 선택은 조직의 기존 기술 스택, 비용 구조, 파트너 생태계, 운영 역량도 함께 봐야 합니다.

### Q3. AWS 서비스가 너무 많은데 어떻게 선택해야 하나요?
- 왜 물을까: 초보자들이 가장 많이 느끼는 어려움
- 짧은 답:
  - 먼저 문제를 `컴퓨팅`, `스토리지`, `데이터베이스`, `네트워크`, `보안`으로 나눠서 생각해야 합니다.
  - 그다음 `내가 직접 관리할 것인지`, `비용이 어떻게 발생하는지`, `유사 서비스와 차이`를 보면 선택이 쉬워집니다.
- 더 들어오면:
  - 예를 들어 컴퓨팅이면 `EC2 vs Lambda vs Fargate`, 데이터베이스면 `RDS vs DynamoDB`처럼 비교 구조로 보면 됩니다.

### Q4. AWS를 쓰면 운영을 전부 AWS가 대신해주나요?
- 왜 물을까: 관리형 서비스에 대한 오해를 확인하려는 질문
- 짧은 답:
  - 아닙니다. AWS는 인프라를 많이 관리해주지만, 권한 설정, 네트워크 설계, 데이터 모델, 애플리케이션 운영은 여전히 사용자가 책임집니다.
- 더 들어오면:
  - 이게 바로 책임 공유 모델이고, 서비스가 관리형일수록 AWS 책임은 늘어나지만 사용자 책임이 사라지지는 않습니다.

## 2. 섹션별 예상질문

### 2.1 클라우드 컴퓨팅이란?

#### Q. CAPEX를 OPEX로 바꾼다는 말은 무슨 뜻인가요?
- 짧은 답:
  - 서버를 미리 많이 사는 대신, 필요한 만큼 사용하고 운영비처럼 지출한다는 뜻입니다.

#### Q. 탄력적 확장이 왜 중요한가요?
- 짧은 답:
  - 트래픽이 늘 때 바로 확장하고 줄 때는 축소해서 성능과 비용을 함께 관리할 수 있기 때문입니다.

#### Q. 속도와 민첩성이 왜 클라우드의 장점인가요?
- 짧은 답:
  - 인프라를 몇 주가 아니라 몇 분 안에 준비할 수 있어서 실험, 배포, 확장이 빨라집니다.

### 2.2 왜 AWS인가?

#### Q. Region과 Availability Zone이 많으면 왜 좋은가요?
- 짧은 답:
  - 사용자와 가까운 위치에 서비스를 둘 수 있고, 장애 대비와 고가용성 설계가 쉬워집니다.

#### Q. AWS의 서비스 수가 많다는 게 왜 장점인가요?
- 짧은 답:
  - 하나의 플랫폼 안에서 컴퓨팅, 스토리지, DB, 보안, 분석 서비스를 연결하기 쉬워집니다.

#### Q. AWS 가격 철학은 발표에서 어떻게 이해하면 되나요?
- 짧은 답:
  - 규모의 경제와 서비스 확장을 통해 더 좋은 선택지와 경험을 제공하려는 방향으로 이해하면 됩니다.

### 2.3 AWS 주요 서비스

#### Q. 서비스 소개에서 가장 먼저 봐야 하는 기준은 무엇인가요?
- 짧은 답:
  - 무엇을 해결하는 서비스인지, 내가 무엇을 관리해야 하는지, 비용이 어떻게 붙는지부터 보면 됩니다.

#### Q. 관리형 서비스가 항상 더 좋은가요?
- 짧은 답:
  - 꼭 그렇지는 않습니다. 운영 부담은 줄지만, 세밀한 제어가 필요한 경우는 EC2 같은 더 직접적인 선택이 맞을 수 있습니다.

### 2.4 AWS 보안

#### Q. AWS가 보안을 책임지면 사용자는 무엇을 신경 써야 하나요?
- 짧은 답:
  - IAM 권한, 보안 그룹, 데이터 접근 제어, 암호화, 공개 설정 등을 사용자가 신경 써야 합니다.

#### Q. IAM, Shield, WAF는 어떻게 다르나요?
- 짧은 답:
  - IAM은 권한 관리, Shield는 DDoS 방어, WAF는 웹 요청 필터링입니다.

## 3. 비교 질문

### Q. EC2와 Lambda의 가장 큰 차이는 무엇인가요?
- 짧은 답:
  - EC2는 서버를 직접 운영하는 방식이고, Lambda는 서버 없이 함수 단위로 실행하는 방식입니다.

### Q. EC2와 Fargate는 어떻게 다른가요?
- 짧은 답:
  - EC2는 서버 관리를 직접 하고, Fargate는 컨테이너는 쓰되 서버 관리는 줄이는 방식입니다.

### Q. S3와 EBS는 어떻게 다른가요?
- 짧은 답:
  - S3는 객체 스토리지이고, EBS는 EC2에 붙여 쓰는 블록 스토리지입니다.

### Q. RDS와 DynamoDB는 어떻게 다른가요?
- 짧은 답:
  - RDS는 관계형 DB이고, DynamoDB는 NoSQL DB입니다.

### Q. Aurora와 일반 RDS는 무엇이 다른가요?
- 짧은 답:
  - Aurora는 RDS 안에서도 성능과 고가용성을 더 강화한 AWS 설계 엔진입니다.

### Q. Region과 Availability Zone은 어떻게 다른가요?
- 짧은 답:
  - Region은 큰 지리적 영역이고, Availability Zone은 그 안의 독립된 데이터센터 단위입니다.

### Q. VPC와 Subnet은 어떻게 다른가요?
- 짧은 답:
  - VPC는 전체 네트워크이고, Subnet은 그 안의 세부 네트워크 구간입니다.

### Q. Shield와 WAF는 어떻게 다른가요?
- 짧은 답:
  - Shield는 DDoS 공격 방어, WAF는 HTTP/HTTPS 요청 필터링입니다.

## 4. 서비스별 예상질문

### Region
#### Q. Region은 왜 중요한가요?
- 짧은 답:
  - 사용자 지연시간, 비용, 규제, 재해복구 전략에 직접 영향을 주기 때문입니다.
#### Q. 여러 Region을 꼭 써야 하나요?
- 짧은 답:
  - 꼭 그렇지는 않지만, 높은 복원력이 필요한 경우 멀티 리전이 유리합니다.

### Availability Zone
#### Q. Availability Zone은 왜 나누어져 있나요?
- 짧은 답:
  - 한 지점 장애가 전체 서비스 장애로 이어지지 않게 하려는 목적입니다.
#### Q. 멀티 AZ는 왜 중요하나요?
- 짧은 답:
  - 같은 리전 안에서 고가용성을 높이는 가장 기본적인 방법이기 때문입니다.

### Subnet
#### Q. Subnet을 왜 나누나요?
- 짧은 답:
  - 자원을 공개 영역과 비공개 영역으로 나누고, 트래픽 흐름을 분리하기 위해서입니다.
#### Q. Public Subnet과 Private Subnet의 차이는 무엇인가요?
- 짧은 답:
  - Public Subnet은 인터넷 접근이 가능하고, Private Subnet은 일반적으로 직접 인터넷에 노출되지 않습니다.

### Amazon VPC
#### Q. VPC는 쉽게 말하면 무엇인가요?
- 짧은 답:
  - AWS 안에서 내가 설계하는 전용 사설 네트워크입니다.
#### Q. VPC가 왜 중요한가요?
- 짧은 답:
  - 보안, IP 구조, 인터넷 연결, 서비스 간 통신을 설계하는 기반이기 때문입니다.

### Amazon EC2
#### Q. EC2는 언제 쓰는 게 가장 적합한가요?
- 짧은 답:
  - 서버 제어권이 필요하거나 특정 OS, 미들웨어, 애플리케이션 설치가 필요한 경우입니다.
#### Q. EC2의 단점은 무엇인가요?
- 짧은 답:
  - OS 패치, 보안, 확장 설정 등 직접 관리할 부분이 많습니다.
#### Q. 발표 슬라이드의 인스턴스 패밀리는 어떻게 설명하면 되나요?
- 짧은 답:
  - 용도별로 CPU, 메모리, 네트워크 비중이 다른 서버 타입이라고 설명하면 됩니다.
#### Q. Reserved Instance와 Savings Plans는 발표에서 어떻게 짧게 말하면 좋나요?
- 짧은 답:
  - 일정 사용량을 약정해서 온디맨드보다 비용을 절감하는 방식이라고 정리하면 됩니다.

### Auto Scaling
#### Q. Auto Scaling은 왜 필요한가요?
- 짧은 답:
  - 트래픽 변화에 따라 서버 수를 자동으로 늘리고 줄여 성능과 비용을 함께 관리하기 위해서입니다.
#### Q. Auto Scaling만 있으면 무중단이 보장되나요?
- 짧은 답:
  - 아닙니다. 여러 인스턴스 구성, 로드 밸런서, 헬스체크 같은 요소와 함께 설계되어야 합니다.

### AWS Lambda
#### Q. Lambda의 가장 큰 장점은 무엇인가요?
- 짧은 답:
  - 서버를 직접 운영하지 않고 이벤트 기반으로 빠르게 실행할 수 있다는 점입니다.
#### Q. Lambda가 모든 컴퓨팅을 대체할 수 있나요?
- 짧은 답:
  - 아닙니다. 장시간 실행, 특수한 서버 환경, 지속 연결이 필요한 워크로드는 다른 서비스가 더 적합할 수 있습니다.

### AWS Fargate
#### Q. Fargate는 누구에게 적합한가요?
- 짧은 답:
  - 컨테이너는 쓰고 싶지만 EC2 서버 관리 부담은 줄이고 싶은 팀에 적합합니다.
#### Q. Fargate와 Lambda는 어떻게 다르나요?
- 짧은 답:
  - Fargate는 컨테이너 실행, Lambda는 함수 실행 중심입니다.

### Amazon S3
#### Q. S3는 왜 많이 쓰이나요?
- 짧은 답:
  - 확장성이 높고 내구성이 좋으며 이미지, 로그, 백업, 정적 파일 등 활용 범위가 넓기 때문입니다.
#### Q. S3는 파일 서버처럼 생각해도 되나요?
- 짧은 답:
  - 완전히 같지는 않습니다. S3는 객체 스토리지이고, 일반 디스크 파일 시스템과 동작 방식이 다릅니다.
#### Q. 발표에 나온 S3 다른 서비스 연계는 왜 중요한가요?
- 짧은 답:
  - S3가 저장소 역할뿐 아니라 데이터 허브처럼 여러 서비스와 연결되기 때문입니다.

### Amazon EBS
#### Q. EBS는 언제 필요하나요?
- 짧은 답:
  - EC2 인스턴스에 지속적인 디스크가 필요할 때 사용합니다.
#### Q. EBS와 S3를 혼동하면 안 되는 이유는 무엇인가요?
- 짧은 답:
  - EBS는 서버 디스크, S3는 파일 저장소라는 점이 근본적으로 다릅니다.

### Amazon CloudFront
#### Q. CloudFront는 왜 쓰나요?
- 짧은 답:
  - 사용자와 가까운 엣지에서 콘텐츠를 제공해 속도를 높이고 원본 서버 부담을 줄이기 위해서입니다.
#### Q. CloudFront는 보안에도 도움이 되나요?
- 짧은 답:
  - 네, WAF나 Shield와 함께 쓰면 성능뿐 아니라 보안 측면에도 도움이 됩니다.

### Amazon Redshift
#### Q. Redshift는 일반 DB와 무엇이 다른가요?
- 짧은 답:
  - 트랜잭션 처리보다 대규모 분석과 집계에 최적화된 데이터 웨어하우스입니다.
#### Q. Redshift를 실시간 서비스 DB로 써도 되나요?
- 짧은 답:
  - 일반적으로는 적합하지 않고, 분석 용도에 더 잘 맞습니다.

### Amazon EMR
#### Q. EMR은 어떤 상황에서 쓰나요?
- 짧은 답:
  - 대용량 데이터를 Hadoop이나 Spark 같은 프레임워크로 처리할 때 사용합니다.
#### Q. EMR이 초보자에게 어려운 이유는 무엇인가요?
- 짧은 답:
  - 프레임워크 이해와 클러스터 운영 개념이 함께 필요하기 때문입니다.

### AWS Data Pipeline
#### Q. Data Pipeline은 어떤 서비스인가요?
- 짧은 답:
  - 데이터를 정기적으로 이동하거나 처리하는 워크플로를 스케줄링하는 서비스입니다.
#### Q. 지금도 많이 쓰이나요?
- 짧은 답:
  - 기존 환경에서는 볼 수 있지만, 신규 설계에서는 Glue나 Step Functions 같은 다른 선택지도 많이 검토합니다.

### Amazon Kinesis
#### Q. Kinesis는 어떤 데이터에 어울리나요?
- 짧은 답:
  - 로그, 클릭스트림, IoT 이벤트처럼 계속 들어오는 실시간 데이터에 적합합니다.
#### Q. Kinesis와 일반 큐는 어떻게 다른가요?
- 짧은 답:
  - Kinesis는 스트리밍 처리 중심이고, 큐는 메시지 전달과 비동기 분리에 더 가깝습니다.

### Amazon SageMaker
#### Q. SageMaker는 누구를 위한 서비스인가요?
- 짧은 답:
  - AWS에서 머신러닝 모델을 학습하고 배포하려는 팀을 위한 서비스입니다.
#### Q. SageMaker가 있으면 ML 인프라를 모두 직접 만들 필요가 없나요?
- 짧은 답:
  - 네, 많은 부분을 관리해주지만 데이터셋, 모델, 학습 로직은 여전히 사용자가 준비해야 합니다.

### PoP
#### Q. PoP는 무엇인가요?
- 짧은 답:
  - 사용자 가까이에서 콘텐츠를 전달하기 위한 엣지 거점입니다.
#### Q. Region과 PoP는 어떻게 다른가요?
- 짧은 답:
  - Region은 서비스가 실행되는 주요 인프라 영역이고, PoP는 콘텐츠 전달용 엣지 위치입니다.

### Amazon RDS
#### Q. RDS는 왜 초보자에게 좋은가요?
- 짧은 답:
  - DB 서버를 직접 설치하고 백업, 패치, 운영하는 부담을 줄여주기 때문입니다.
#### Q. RDS가 있으면 DB 운영을 하나도 안 해도 되나요?
- 짧은 답:
  - 아닙니다. 스키마, 쿼리, 성능, 계정, 접근 통제는 여전히 신경 써야 합니다.
#### Q. 발표 슬라이드의 EC2 위 DB와 RDS 비교는 어떻게 말하면 되나요?
- 짧은 답:
  - EC2 위 DB는 직접 운영, RDS는 관리형 운영이라고 정리하면 됩니다.

### Amazon Aurora
#### Q. Aurora는 왜 따로 강조되나요?
- 짧은 답:
  - RDS 계열 중에서도 성능과 가용성 측면에서 강점을 가진 엔진이기 때문입니다.
#### Q. Aurora가 항상 RDS보다 좋은가요?
- 짧은 답:
  - 항상은 아닙니다. 요구사항과 비용, 호환성에 따라 일반 RDS가 더 적합할 수도 있습니다.

### Amazon DynamoDB
#### Q. DynamoDB는 언제 적합한가요?
- 짧은 답:
  - 매우 빠른 응답과 큰 규모의 확장이 필요한 NoSQL 워크로드에 적합합니다.
#### Q. DynamoDB를 관계형 DB처럼 쓰면 안 되는 이유는 무엇인가요?
- 짧은 답:
  - 설계 기준이 조인 중심이 아니라 접근 패턴 중심이기 때문입니다.

### Amazon ElastiCache
#### Q. ElastiCache는 왜 필요한가요?
- 짧은 답:
  - 자주 조회되는 데이터를 메모리에 두어 응답 속도를 높이고 원본 DB 부하를 줄이기 위해서입니다.
#### Q. 캐시를 쓰면 데이터베이스가 필요 없나요?
- 짧은 답:
  - 아닙니다. 캐시는 보조 계층이고, 원본 데이터 저장소를 대체하지 않습니다.

### Security Group
#### Q. Security Group은 쉽게 말하면 무엇인가요?
- 짧은 답:
  - EC2 같은 자원 앞단에서 트래픽을 제어하는 가상 방화벽입니다.
#### Q. Security Group과 IAM은 왜 다르나요?
- 짧은 답:
  - Security Group은 네트워크 접근 제어이고, IAM은 사용자와 서비스 권한 제어입니다.

### Amazon IAM
#### Q. IAM의 핵심 역할은 무엇인가요?
- 짧은 답:
  - 누가 어떤 AWS 자원에 어떤 작업을 할 수 있는지 통제하는 것입니다.
#### Q. IAM에서 가장 중요한 원칙은 무엇인가요?
- 짧은 답:
  - 필요한 최소 권한만 주는 least privilege 원칙입니다.

### AWS Shield
#### Q. Shield는 무엇을 막아주나요?
- 짧은 답:
  - DDoS 공격 같은 대량 트래픽 기반 공격으로부터 서비스를 보호합니다.
#### Q. Shield만 있으면 웹 보안이 끝나나요?
- 짧은 답:
  - 아닙니다. 애플리케이션 계층 공격 대응에는 WAF 같은 서비스가 함께 필요합니다.

### AWS WAF
#### Q. WAF는 어떤 공격을 막는 데 유용한가요?
- 짧은 답:
  - 비정상적인 웹 요청, 일부 취약점 공격 패턴, 악성 봇 트래픽 등에 대응할 때 유용합니다.
#### Q. WAF에서 가장 조심해야 할 점은 무엇인가요?
- 짧은 답:
  - 규칙을 너무 강하게 잡으면 정상 요청도 차단할 수 있다는 점입니다.

## 5. 조금 더 들어올 수 있는 꼬리질문

### Q. 무중단 운영을 위해 발표에서 가장 간단하게 강조할 포인트는 무엇인가요?
- 짧은 답:
  - 멀티 AZ, 로드 밸런서, Auto Scaling, 관리형 서비스를 조합하면 무중단에 가까운 구조를 만들기 쉽다고 말하면 됩니다.

### Q. AWS에서 비용이 많이 나오는 대표 원인은 무엇인가요?
- 짧은 답:
  - 항상 켜져 있는 큰 EC2, 과도한 스토리지, 데이터 전송, 불필요한 관리형 자원 유지가 대표적입니다.

### Q. 초보자가 가장 먼저 실수하는 AWS 보안 문제는 무엇인가요?
- 짧은 답:
  - 과도한 IAM 권한, 너무 넓은 Security Group, 잘못된 공개 설정이 가장 흔합니다.

### Q. 발표 중 “이 서비스가 꼭 필요한가요?”라는 질문을 받으면 어떻게 답하나요?
- 짧은 답:
  - 모든 서비스가 필수는 아니고, 문제 유형에 따라 적절한 서비스를 선택하는 것이 AWS의 핵심이라고 답하면 됩니다.

## 6. 발표 직전 암기 포인트

- 클라우드의 핵심 키워드: `민첩성`, `탄력성`, `비용 구조 변화`
- AWS 선택 이유 키워드: `글로벌 인프라`, `서비스 다양성`, `운영 경험`, `생태계`
- 컴퓨팅 비교: `EC2 = 서버 직접 관리`, `Lambda = 서버리스 함수`, `Fargate = 서버리스 컨테이너`
- 스토리지 비교: `S3 = 객체`, `EBS = 블록`
- DB 비교: `RDS = 관계형`, `DynamoDB = NoSQL`, `Aurora = 고성능 관리형 관계형`
- 보안 비교: `IAM = 권한`, `Security Group = 네트워크 접근`, `Shield = DDoS`, `WAF = 웹 요청 필터`
